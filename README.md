# TechAtlas

> **Discover Engineering. Faster.**

TechAtlas is a backend-first engineering knowledge search engine that aggregates content from multiple trusted technical sources into a unified search experience. Instead of searching across several websites individually, TechAtlas provides a single interface for discovering engineering knowledge with fast, relevant, and explainable search results.

The project is built from scratch to demonstrate the core principles behind modern search engines, including information retrieval, indexing, ranking, and scalable backend architecture.

---

## ✨ Features

* Unified search across multiple engineering knowledge sources
* BM25-based document ranking
* Custom inverted index implementation
* Incremental indexing for efficient updates
* RESTful API built with Spring Boot
* Autocomplete using a Trie data structure
* Search analytics and query tracking
* Extensible architecture for adding new content sources

## 🛠️ Getting Started (Phase 1 Setup)

### Prerequisites
- **Java 21**
- **Maven 3.9+**
- **Docker & Docker Compose**

### Running the Application

1. **Start the database container**:
   ```bash
   docker compose up -d
   ```
   This launches a PostgreSQL 16 container bound to local port `5432` with user `techatlas` and database `techatlas`.

## Local Setup

Create a file:

src/main/resources/application-local.yml

using:

src/main/resources/application-local.example.yml

and configure your local PostgreSQL credentials.

2. **Build the application**:
   ```bash
   mvn clean install
   ```
   This will verify compilation, run the checkstyle/validation plugins (if configured), and execute unit/integration tests with an in-memory database profile.

3. **Run the application**:
   ```bash
   mvn spring-boot:run -Dspring-boot.run.profiles=local
   ```
   This runs the application using the `local` profile, connecting it to the Docker PostgreSQL database.

### Verifying endpoints
- **Swagger UI**: Accessible at `http://localhost:8080/swagger-ui.html`
- **Health Check**: `GET http://localhost:8080/api/v1/health` (should return `{"status": "UP"}`)

## 💾 Phase 2: Persistence Layer Architecture

We have established a robust persistence layer to store, retrieve, update, and delete search documents.

### Entity Overview (`Document`)
The domain model is centered around the `Document` class, containing the following properties:
- **`id`**: Unique `UUID` generated by the persistence provider.
- **`title`**: String (up to 255 chars), cannot be blank.
- **`content`**: Text containing document body, cannot be blank.
- **`url`**: String (up to 2048 chars) pointing to the original document location, must be a valid URL.
- **`source`**: Enum value representing the origin (`WIKIPEDIA`, `GITHUB`, `STACKOVERFLOW`, `REDDIT`, `MANUAL`).
- **`category`**: Optional string category (up to 100 chars).
- **`author`**: Optional string author (up to 255 chars).
- **`language`**: Optional string language (up to 50 chars).
- **`contentHash`**: Auto-calculated SHA-256 hash of the content to enforce uniqueness and detect duplicates.
- **`status`**: Enum representing current status (`ACTIVE`, `DELETED`, `PENDING_INDEX`, `FAILED`).
- **`createdAt` / `updatedAt`**: Non-null timestamps managed automatically via `@CreationTimestamp` and `@UpdateTimestamp`.
- **`indexedAt`**: Nullable timestamp indicating when the document was indexed in search indices.
- **`metadata`**: Optional text block storing provider-specific attributes.

### Database Design & Migrations
- Managed via Flyway in script `V2__create_documents_table.sql`.
- Key database columns map directly to the `Document` JPA entity.
- The `documents` table is indexed on the columns `source`, `status`, `category`, and `content_hash` to ensure fast query lookups.
- A unique constraint is placed on `content_hash` to guarantee data integrity at the database level.

### REST Endpoints
The following endpoints manage documents:
- **`POST /api/v1/documents`**: Create a new document. Computes content hash and defaults status to `PENDING_INDEX`. Returns `201 Created`.
- **`GET /api/v1/documents`**: Retrieves all stored documents. Returns `200 OK`.
- **`GET /api/v1/documents/{id}`**: Retrieves a specific document by its UUID. Returns `200 OK` or `404 Not Found`.
- **`PUT /api/v1/documents/{id}`**: Updates an existing document. Recalculates content hash and ensures unique integrity. Returns `200 OK`, `404 Not Found`, or `409 Conflict`.
- **`DELETE /api/v1/documents/{id}`**: Deletes a specific document. Returns `204 No Content` or `404 Not Found`.

All controllers return strictly JSON (never HTML). Bean Validation detects and rejects malformed requests, returning clean error messages via the unified global exception handler.
